1. ABSTRACT
================================================================================

Text-to-image diffusion models such as Stable Diffusion 1.5 are pretrained on
large web-scale corpora but often fail on compositional prompts (object counts,
attributes, spatial relations, surreal juxtapositions). Human-feedback reward
models and reinforcement learning (RL) fine-tuning are a promising way to align
generation with such constraints without retraining from scratch.

This project began as an exploration of *structured*, *hierarchical* rewards
built from scene graphs and contrastive CLIP scoring ("Diffusion Graphs"). The
research trajectory then incorporated DPOK (Diffusion Policy Optimization with
KL regularization) — an online policy-gradient method from Fan et al. (2023) —
as the primary fine-tuning engine, using ImageReward as the training signal
while retaining hierarchical reward code and rich offline evaluation (VQA, DSG,
T2I-CompBench-style decomposition) for analysis and future scene-graph variants.

We implemented a self-contained, paper-faithful DPOK training loop on Bocconi HPC
(NVIDIA H100 NVL, 80 GB), fine-tuning SD1.5 via LoRA on the UNet attention
projections, with importance sampling, PPO-style ratio clipping, KL anchoring to
a frozen reference UNet, optional value-function baseline, and Classifier-Free
Guidance (CFG) applied consistently in sampling and policy log-probability.

Empirical highlights:
  • Single-prompt regime ("A green colored rabbit.") shows strong ImageReward
    gains (+1.75 mean delta vs baseline on 30 seeds; training snapshots rise from
    ~−1.6 to ~+1.5 IR over ~1500 rounds).
  • Four-prompt paper reproduction (Sec 5.2) yields modest aggregate gains on
    training prompts and no generalization on held-out prompts with the same
    compositional axes.
  • Extensive hyperparameter debugging documented pathologies (over-tight IS
    clip, tiny effective learning rate, gradient checkpointing vs PEFT, fp32 OOM).

  [See Appendix D for the full figure catalog and where to place each plot in
   the written report. Key result figures: Fig. 5 baseline_vs_trained_grid,
   Fig. 2 per_prompt_curves, Fig. 3 snapshot_grid.]

The codebase cleanly separates a production DPOK pipeline (repository root) from
earlier research scaffolding (`src/`) that informed reward design but is not
imported by training scripts.

2. RESEARCH QUESTION AND OBJECTIVES
================================================================================

2.1 Core research question
--------------------------

Can we improve compositional text-to-image generation by fine-tuning Stable
Diffusion with (a) online RL (DPOK) rather than only inference-time steering,
and (b) rewards that respect prompt structure — either via human-preference
models (ImageReward) or via hierarchical scene-graph decompositions (CLIP on
objects, attributes, relations)?

This question sits at the intersection of:
  • Alignment / RLHF for generative models (reward maximization under KL trust
    region to a reference policy).
  • Structured visual semantics (scene graphs from COCO/OpenPSG/GQA).
  • Compositional generalization (does fine-tuning on four curated prompts
    transfer to four held-out prompts with parallel structure?).

2.2 Novelty and real-world relevance (rubric: "interesting research question")
-------------------------------------------------------------------------------

Novelty in this repository is primarily *engineering and methodological integration*
rather than a new RL algorithm:

  • Combining DPOK-style online fine-tuning with a long-running hierarchical
    reward research line (contrastive CLIP gates, message-passing graphs, FK
    particle steering) in one coherent codebase.
  • Deliberate train/eval split: cheap dense reward at training time (ImageReward),
    expensive structured metrics offline (VQA, DSG, decomposed CLIP).
  • Paper-faithful IS + CFG-matched policy gradient in a single auditable file,
    with diagnostic tooling (fixed-seed per-prompt snapshots, W&B logging).

Real-world relevance: compositional control matters for advertising, design
tools, robotics simulation, and any domain where prompts specify multiple
entities and relations. RL fine-tuning with learned rewards is increasingly
used in commercial T2I stacks; understanding failure modes (reward hacking,
lack of holdout generalization, scale mismatch vs published baselines) is
directly applicable.

3. BACKGROUND AND RELATED WORK
================================================================================

3.1 DPOK (Fan et al., 2023)
---------------------------

DPOK treats text-to-image diffusion sampling as a Markov decision process:
  • State: latent x_t at denoising step t, plus conditioning (text embedding z).
  • Policy: the denoising transition p_θ(x_{t-1} | x_t, z) induced by the UNet.
  • Reward: scalar feedback on the final decoded image (e.g., ImageReward).

The training objective combines:
  • Policy gradient term weighted by advantage (reward minus optional baseline),
    with importance sampling (IS) ratios between current and behavior policy,
    and PPO-style clipping (eq. 26 in the paper).
  • KL regularization toward a frozen *reference* policy π_ref (pretrained SD),
    preventing collapse and preserving diversity (eq. 9).

The paper argues online RL outperforms supervised fine-tuning on the reward for
both image-text alignment and perceptual quality, with careful KL analysis.

Official code: Google Research `dpok` repository (multi-file: extended pipeline,
extended DDIM scheduler, `train_online_pg.py`). Our implementation consolidates
equivalent logic into one script for auditability.

3.2 ImageReward
---------------

ImageReward (Xu et al., 2023) is a human-preference model trained to score
(image, prompt) pairs. It provides a wider dynamic range than raw CLIP cosine
similarity (~[−3, +3] vs narrow cosine bands), improving signal-to-noise for RL.
DPOK paper experiments use ImageReward as the primary reward.

3.3 Hierarchical and structured rewards (project lineage)
---------------------------------------------------------

Prior work in this codebase (under `src/`) includes:
  • `SimpleHierReward`: decomposes a hand-built graph into nodes (existence),
    bindings (attributes), and interactions (relations); gates binding/interaction
    terms by node existence; contrastive negatives per component.
  • `RewardGraph`: message-passing variant propagating gates along edges.
  • FK (Feynman-Kac) steering: particle-based inference-time guidance without
    weight updates — complements DPOK (learning vs steering).

Structured offline metrics:
  • DSG-style atomic VQA claims (regex decomposition in our code).
  • T2I-CompBench-style object/attribute/relation buckets.
  • CLIPScore (ViT-L/14) for global alignment.

3.4 Datasets (scene graphs)
---------------------------

  • COCO captions: plain text prompts (used in production).
  • OpenPSG: panoptic scene graphs with relations (explored in notebooks/analysis;
    not used in current ImageReward DPOK runs).
  • GQA: question-answering scene graphs (idem).

Data analysis notebooks (`data_analysis/gqa.ipynb`, `openpsg.ipynb`) produced
summary JSON artifacts for understanding graph statistics; they informed prompt
design but are not on the hot training path.

4.1 Original research direction: "Diffusion Graphs"
---------------------------------------------------

The project name `simple-hier-clip-reward` reflects the initial core idea:
score generated images with a *hierarchical* reward derived from a *scene graph*
and CLIP (OpenCLIP ViT-L/14), rather than a single global cosine similarity.

Starting components (still present under `src/`):

  src/rewards/clip.py
    CLIPScorer with variable-resolution positional embedding interpolation —
    supports gradient flow through non-standard image sizes (important if
    optimizing latents or doing analysis).

  src/rewards/rewards.py
    Dataclasses Node, Binding, Interaction, Graph; SimpleHierReward and RewardGraph
    modules with soft/hard gating and contrastive scoring modes (`prob` vs `margin`).
    Default hierarchical weights (when all components present): nodes 0.30,
    bindings (attributes) 0.35, interactions (relations) 0.25, global CLIP 0.10.
    Gating: soft sigmoid α·(φ−τ) with τ=0.5 or hard threshold; contrastive negatives
    per node/binding/interaction penalize wrong phrases (inherits into later
    antonym-based designs).

  src/refgraphs.py, src/refgraphs2.py
    Hand-written reference graphs (subjects, clothing, animals, glasses) and

  src/simple_hier_clip_reward.py, src/message_passing_clip_reward.py
    Driver scripts comparing graph variants on fixed demo images.

  src/example.py, src/example_t2i.py, src/example_fksteering.py
    Prototypes chaining SD1.5 sampling with reward feedback; FK steering variant

  src/fk_steering/fkd_diffusers/
    FKD class, SD pipeline subclass, reward registry (CLIP, ImageReward, project rewards).

  src/viz/viz_graph.py
    Graphviz rendering of which graph components contributed most to the reward.

4.2 What was NOT working as a complete training system initially
----------------------------------------------------------------

The `src/` tree demonstrated *how to score* images and *how to steer* at
inference time, but did not provide a stable, paper-aligned *online RL training
loop* with:
  • LoRA fine-tuning of UNet under KL constraint,
  • Trajectory storage and importance sampling,
  • Human-preference reward at scale on HPC,
  • Rigorous eval protocol matching DPOK paper tables.

4.3 Transition to DPOK production pipeline
------------------------------------------

A prior script generation (`dpok_hpc_wandb.py`, since removed) bridged toward
cluster training. The current production entry point is `dpok_imagereward.py`,
which:
  • Drops `src/` imports entirely for training reliability on HPC.
  • Uses ImageReward instead of CLIP-ViT-B-32 cosine for RL signal.
  • Implements Appendix B online RL hyperparameters with extensive CLI flags.

`dpok_scene_reward.py` was written as a self-contained port of data loaders +
RewardModel / HierarchicalRewardModel / BLIPRewardModel for a future variant
where the RL reward is scene-graph-aware; it mirrors ideas from `src/` but is
not wired into current ImageReward jobs.

4.4 Scene-graph reward variants at repository root (not in src/)
---------------------------------------------------------------

Beyond `src/` and `dpok_scene_reward.py`, two root modules complete the
"structured reward" research line:

  perturb_graph.py
    Graph-only perturbation harness for sensitivity analysis. Functions:
    perturb_remove_object, perturb_swap_attribute (hand-curated color/size
    antonym tables), perturb_swap_spatial_relation (left↔right, etc.),
    perturb_swap_unrelated_noun. Used by notebooks/sg_demo_real.ipynb to build
    perturbation grids (Fig. 15) without touching images — tests whether a
    reward drops when the graph is wrong.

  dpok_sg_reward.py
    "SG-aware" BLIP-ITM reward: scores whole image against per-object,
    per-attribute (with antonym contrast), and per-relation phrases derived
    from SceneGraph; small global ImageReward term. Designed as a DPOK-ready
    alternative to flat ImageReward when scene graphs are available. Not used
    in jobs 484796/798 but documents the intended integration path.

  notebooks/sg_demo_real.ipynb
    End-to-end demo on four FIBO-released (image, structured-caption) examples:
    builds SceneGraphs, runs perturb_graph variants, caches SD outputs, plots
    real_vs_sd_orig and perturbation grids. Connects OpenPSG/GQA-style structure
    to human-readable prompts without requiring full datasets on disk.

6. METHODS — DPOK ALGORITHM (THEORY AND OUR INSTANTIATION)
================================================================================

6.1 Problem formulation
-----------------------

Given pretrained SD1.5 parameters θ_0, we learn LoRA adapters Δθ on the UNet
attention layers so that the induced policy π_θ maximizes expected terminal
reward R(x_0, z) where z is the text prompt embedding and x_0 is the final
latent after T denoising steps.

A frozen copy of the base UNet serves as π_ref for KL penalization.

6.2 Sampling (behavior policy)
------------------------------

Algorithm: DDIM-style scheduler with η=1.0 (stochastic; paper's DDPM-equivalent
choice). Default T=20 steps in our runs (paper uses 50 — faster rollouts, fewer
transitions per trajectory).

Classifier-Free Guidance (CFG): scale s=7.5 applied during trajectory collection
by duplicating latents, running UNet on concatenated [uncond, cond] embeddings,
and combining noise predictions as ε_u + s(ε_c − ε_u).

Each trajectory stores for every step t:
  (x_t, x_{t-1}, t, noise_pred_old)
where noise_pred_old is the *CFG-combined* prediction used to produce x_{t-1}.

`sample_trajectory()` is wrapped in `@torch.no_grad()` — sampling is not
differentiable; gradients enter only during the policy update on stored pairs.

6.3 Policy log-probability and importance sampling
--------------------------------------------------

The transition kernel is modeled as Gaussian:
  p_θ(x_{t-1} | x_t) = N(x_{t-1}; μ_θ(x_t, t), σ_t² I)

with μ_θ obtained linearly from the UNet noise prediction (DDIM mean formula).
σ_t² from DDIM posterior variance with floor SIGMA_FLOOR=1e-5.

For a stored transition, the log-ratio:
  log ρ = log p_θ(x_{t-1}|x_t) − log p_θ_old(x_{t-1}|x_t)

Because variances match, constants cancel:
  log ρ = (||x_{t-1} − μ_old||² − ||x_{t-1} − μ_new||²) / (2σ_t²)

PPO-style clipped surrogate uses:
  clip(ρ, 1−ε, 1+ε)  with ε = `--is_clip` (default 1e-4 paper; 1e-3 in production runs).

Policy loss term (per pair):
  L_policy = − α · Â · clip(ρ, 1−ε, 1+ε)
where Â is advantage (reward or reward − V).

6.4 KL regularization
---------------------

We penalize deviation of the *noise prediction* from the reference UNet:
  L_KL = β · ||ε_θ − ε_ref||² / (2σ_t²)

Official Google code uses unnormalized squared L2 on noise; we divide by σ_t² so
β aligns with Gaussian KL units. Both are defensible; ours tracks the analytic
derivation in the paper's eq. 9 more directly.

6.5 Terminal reward
-------------------

ImageReward-v1.0, frozen, applied to VAE-decoded PIL image and prompt string.
Rewards clipped to [−5, 5] before loss. Optional value network predicts V(x_t,t,z)
from latent + sinusoidal time embedding + pooled text embedding; MSE to realized
r; advantage uses V detached from policy gradient path.

6.6 Training loop structure (our design vs official)
----------------------------------------------------

Per round:
  1. Sample m=10 trajectories (random prompts from list).
  2. Score terminal images with ImageReward.
  3. Pool all (x_t, x_{t-1}, t) pairs (exclude final step without x_{t-1}).
  4. For K=5 gradient steps:
       - Sample n=`--is_batch` pairs with replacement (paper n=32; we often use 8).
       - Average IS+KL(+VF) loss, backward, grad clip, AdamW step on LoRA.
  5. Log means; every `save_every` rounds checkpoint + fixed-seed snapshots.

Differences from official DPOK (see MEETING_WALKTHROUGH.md §3):
  • Single-round on-policy buffer vs FIFO 1000-transition off-policy buffer.
  • No gradient accumulation micro-batching (we sum n pairs in one backward).
  • Joint VF optimizer step vs separate value-training phase.
  • PEFT LoRA vs deprecated LoRACrossAttnProcessor.
  • No gradient checkpointing (PEFT incompatibility caused zero LoRA-B grads).

6.7 Critical implementation bug fix
-----------------------------------

Early versions applied `@torch.no_grad()` to `noise_pred_to_mu`, which zeroed
gradients through μ_θ and made the policy loss non-informative. Removing no_grad
on that path was essential for learning signal.

6.8 Model and optimizer details
-------------------------------

  • Base: runwayml/stable-diffusion-v1-5, safety checker disabled.
  • Trainable: LoRA rank r=4 (paper default) on to_q, to_k, to_v, to_out.0;
    lora_alpha=r → scaling α/r = 1. ~797K trainable params (~0.09% of UNet).
  • Optimizer: AdamW on LoRA params, lr=1e-5 paper → 1e-4 in tuned runs.
  • Grad norm clip: 0.1 paper → 1.0 tuned.
  • VAE decode in fp32 for numerical stability when pipe uses bf16/fp32.
  • Diagnostics: `is_finite_tensor` guards; `zero_loss()` if all pairs skipped;
    first-step LoRA-B gradient nonzero check.

7.1 Training reward: ImageReward
--------------------------------

Used exclusively in `dpok_imagereward.py` for RL signal. Rationale: human

7.2 Alternate training rewards (implemented, unused in current runs)
--------------------------------------------------------------------

In `dpok_scene_reward.py`:

  RewardModel — global CLIP ViT-L/14 cosine, scaled: (sim − baseline) × scale.
  BLIPRewardModel — BLIP-ITM matching probability mapped to roughly ±5.
  HierarchicalRewardModel — weighted mix of global, per-object, per-relation,
    per-attribute CLIP scores with auto-renormalized weights when graph fields
    missing.

7.3 Offline evaluation metrics (`dpok_eval_metrics.py`)
---------------------------------------------------------

  CLIPScorer (ViT-L-14 OpenAI weights) — global text-image cosine.
  ImageRewardScorer — same model as training reward; used for held-out comparison.
  VQAScorer (t2v-metrics, clip-flant5-xl) — P("Does this image show <prompt>?").
  DSGScorer — mean VQA over regex-extracted atomic claims from prompt.
  T2ICompScorer — VQA split into object / attribute / relation question buckets.

`eval_report.py` orchestrates image generation (baseline SD vs LoRA) and writes:
  report.txt, report.json, report.tex, per_image_scores.csv, distribution plots,
  radar chart, per-prompt breakdown tables.

Default eval protocol for paper-style JSON prompts: 30 seeds per prompt × N prompts
(120 images per side for 4 prompts).

7.4 Structured metric decomposition (DSG and T2I-CompBench-style)
-----------------------------------------------------------------

Our eval stack does not call external LLMs for decomposition (unlike full DSG
paper pipelines). Instead:

  extract_atomic_claims() in dpok_eval_metrics.py
    Rule-based prompt → yes/no questions, e.g. "Is there a rabbit?",
    "Is the rabbit green?", "Is the dog on the moon?" Used by DSGScorer as
    mean VQA probability over claims (lightweight DSG proxy).

  generate_t2i_questions() in dpok_eval_metrics.py
    Same prompt split into object / attribute / relation buckets for
    T2ICompScorer (optional via --metrics ...,t2i in eval_report.py).

Rationale: offline eval can afford VQA forward passes; training cannot afford
per-claim VQA at every RL step. DSG gains (+1.7% single-prompt) lag ImageReward
(+457% relative on same run) — evidence of reward–metric mismatch (§13.2).

7.5 Scene-graph BLIP reward (`dpok_sg_reward.py`) — planned training signal
---------------------------------------------------------------------------

Pipeline summary (for future DPOK integration):
  • Object nodes: BLIP-ITM P("a photo of <obj>" | image).
  • Attributes: BLIP(positive phrase) − BLIP(antonym phrase) using tables from
    perturb_graph.py.
  • Relations: BLIP(relation phrase) − BLIP(spatial antonym).
  • Global: small ImageReward on stringified graph.

Sensitivity properties documented in module docstring: remove_object lowers mean;
swap_attribute flips contrast sign. Trade-off vs hierarchical CLIP in src/:
no region crops, but antonym contrast remains the main discriminator.

10. HYPERPARAMETER TUNING AND DEBUGGING HISTORY
================================================================================

Documented in additional/HANDOFF.md. Summary:

Run 482909 — paper defaults
  alpha=10, beta=0.01, lr=1e-5, grad_clip=0.1, is_clip=1e-4, bf16, lora_rank=4
  Outcome: kL stuck ~3e-4; ~50% IS gradient elements clipped; flat mean_r.

Run 483053 — loosen IS clip, reduce alpha
  is_clip=1e-3, alpha=5
  Outcome: clipping rate dropped to ~5%, but kL still flat — policy not moving.

Run 483271/273 — higher lr, grad_clip 1.0, fp32, lora_rank 16
  Outcome: 483273 OOM on 80 GB (fp32 + is_batch=32 + many simultaneous graphs).

Runs 484796 / 484797 / 484798 — current production recipe
  alpha=5, beta=0.01, lr=1e-4, grad_norm_clip=1.0, is_clip=1e-3,
  is_batch=8 (VRAM), pipe_dtype fp32, lora_rank=4,
  use_value_function, vf_lr=1e-4, vf_weight=0.5,
  PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True

Interpretation:
  • Tight ε=1e-4 made clip band [0.9999, 1.0001] — destroyed policy gradient.
  • Small lr × aggressive grad clip → effective step ~1e-6.
  • fp32 stabilizes small LoRA gradients but needs smaller IS batch.
  • Single-prompt concentrates samples on one reward landscape → learns;
    multi-prompt dilutes budget → weaker per-prompt movement.

Reward scale mystery: paper reports rabbit baseline IR ~+0.84; we measure ~−0.38
with same model name — possible normalization difference (official code uses
(mean, std) normalization on rewards).

11.1 Independent variables
--------------------------
  • Prompt regime: single vs 4 paper vs ~104 COCO.
  • Training budget: total_samples (10K four-prompt job; 15K single-prompt script).

  • Training: mean_r per round, kL (KL term magnitude), IS clip rate, ratio stats,

11.4 Controls and fairness
--------------------------
  • CFG=7.5 consistent across train sampling, policy loss, snapshots, eval.
  • Fixed seeds per prompt index for snapshot grid (isolate LoRA change).
  • Eval uses separate conda env to avoid training dependency conflicts.
  • 30 seeds per prompt matches paper's multi-sample reporting spirit.

11.5 What was not fully ablated
-------------------------------
  • Six hyperparameter changes applied simultaneously in v3 ("throw everything"
    strategy) — causality per knob not isolated.
  • T=20 vs T=50 DDIM steps not sweeped systematically in final runs.

13. DISCUSSION
================================================================================

This section interprets §12. Each subsection below states which figure(s) to
cite when making the corresponding argument in the written report.

13.1 What worked
----------------
  • End-to-end DPOK pipeline on HPC with reproducible SLURM scripts and rich logging.
  • Single-prompt fine-tuning clearly increases ImageReward and VQA metrics.
  • Importance sampling implementation verified (LoRA-B gradients nonzero; kL growth).
  • Evaluation harness separates cheap/global vs expensive/structured metrics.

  Discussion + figures:
    Support the single-prompt success claim with Fig. 2 (single-prompt curve),
    Fig. 3 (rabbit row shows visible compositional shift), Fig. 5a (side-by-side
    samples), and Table 1. Mention Fig. 7a VQA histogram alongside Table 1 to
    show alignment gains beyond ImageReward alone.

    Support "IS implementation verified" with Fig. 4 (non-flat segments) and/or
    Fig. 18 W&B kL panel (kL grows from ~1e-6 toward ~1e-4 in job 484796).

13.2 What partially worked
---------------------------
  • Four-prompt training: small aggregate gains; two prompts (wolves, moon) barely
    change; rabbit under multi-prompt lags single-prompt regime.
  • CLIP and DSG improvements are modest — RL optimized ImageReward, not CLIP/DSG
    directly (reward mismatch).

  Discussion + figures:
    Compare Fig. 2a (single) vs Fig. 2b (multi) on the rabbit prompt — same
    algorithm, different prompt budget sharing. Use Fig. 3b rabbit row (flat IR)
    vs Fig. 3 single run (rising IR) as the central visual for "prompt competition."

    Use Fig. 7b: CLIP/DSG histograms barely separate trained from baseline while
    ImageReward mode shifts — classic reward mismatch story. Fig. 6 radar optional.

13.3 What did not work
----------------------
  • Held-out prompt generalization at aggregate level — suggests overfitting to
    training prompt distribution or insufficient capacity/steps per axis.
  • Matching paper absolute ImageReward scales — complicates direct numeric comparison.
  • Scene-graph hierarchical reward integrated into DPOK training — deferred.

  Discussion + figures:
    Place Fig. 5c and Fig. 7c (holdout) immediately after stating generalization
    failure. Fig. 8 bar chart (holdout IR/VQA deltas ≤ 0 for most prompts) is the
    clearest single figure for this subsection.

    For deferred hierarchical reward, cite Fig. 12–16 (§4 starting point) as
    "planned extension" without claiming DPOK results on graph rewards.

13.4 Engineering trade-offs validated
---------------------------------------
  • Flat reward at train time, structured metrics at eval time — necessary for GPU
    hours; hierarchical reward would multiply forward passes per trajectory.
  • Single-file DPOK vs official multi-file — improved auditibility at cost of
    feature parity (no off-policy buffer).
  • PEFT LoRA without checkpointing — memory vs correct gradients.

  Discussion + figures:
    Fig. 1 (pipeline diagram) supports the train/eval split design choice.
    Fig. 18 (W&B from run 482909 vs 484796) supports hyperparameter/debugging
    narrative in §10 if moved here as a "lessons learned" panel.
    Fig. 17 (FK steering) supports comparison "inference steering vs weight update"
    as alternative alignment path (no learning persistence across calls).

13.5 Conclusions (main takeaways)
---------------------------------

  1. DPOK + ImageReward + LoRA on SD1.5 is implementable and can improve
     compositional alignment when the online sample budget focuses on a single
     prompt (strong IR and VQA gains; visible snapshot progression).

  2. Multi-prompt paper reproduction shows weaker, prompt-heterogeneous learning
     and no aggregate generalization to held-out prompts with parallel structure —
     important negative result for "four prompts are enough" hypotheses.

  3. The repository preserves two coherent research layers: (a) production DPOK
     with flat human-preference reward, (b) structured graph rewards and
     perturbation tooling for finer-grained supervision — ready for integration
     but not yet combined in one training run.

  4. Engineering lessons (IS clip, effective learning rate, PEFT vs checkpointing,
     train/eval CFG match, dual conda envs) are as valuable as the headline metrics
     for future RL-on-diffusion work.

14. LIMITATIONS AND OPEN ISSUES
================================================================================

  1. Reward scale / normalization vs official DPOK codebase not fully reconciled.
  2. DDIM steps T=20 vs paper T=50 — fewer transitions per trajectory.
  3. is_batch=8 vs paper n=32 — lower gradient variance per step.
  4. Multi-prompt sample efficiency — 10K samples split across 4 prompts vs 15K
     on one prompt.
  5. No systematic human evaluation (only automated metrics).
  6. Value function benefit unclear; adds parameters and compute.
  7. Potential reward hacking — high ImageReward with limited DSG gain on rabbit.
  8. Held-out failure may indicate memorization of color/object co-occurrences.
  9. Missing requirements.txt and README in repo — versions and install steps
     exist only in setup_cluster.sh / setup_eval_env.sh (see §16.5).
 10. 104-prompt COCO run (484797) outcomes may not be present in local eval dumps.
 11. No human subjective evaluation or A/B user study — all conclusions from
     automated metrics (CLIP, ImageReward, VQA, DSG).
 12. T2I-CompBench-style per-category scores implemented but not in default
     eval runs (clip,imagereward,vqa,dsg only).

APPENDIX C — KEY EQUATIONS
================================================================================

DDIM mean (schematic):
  μ_θ(x_t, t) = (1/√α_t) · ( x_t − (β_t/√(1−ᾱ_t)) · ε_θ(x_t,t) )

Gaussian log-prob up to constant:
  log p_θ(x_{t-1}|x_t) ∝ −||x_{t-1} − μ_θ||² / (2σ_t²)

Importance ratio:
  ρ = exp( log p_θ(x_{t-1}|x_t) − log p_θ_old(x_{t-1}|x_t) )

Clipped policy loss (per transition):
  L_π = − α · Â · clip(ρ, 1−ε, 1+ε)

KL penalty (our σ-normalized form):
  L_KL = β · ||ε_θ − ε_ref||² / (2σ_t²)

Total loss:
  L = L_π + L_KL + λ_VF · (V − r)²

Advantage:
  Â = r − stop_grad(V)   if value function enabled
  Â = r                  otherwise

APPENDIX E — COMPARISON TO OFFICIAL GOOGLE DPOK CODE
================================================================================

Condensed from additional/MEETING_WALKTHROUGH.md §3. Full line-level mapping
lives in that document for supervisor review.

| Aspect | Official DPOK (google-research/dpok) | This repository | Equivalence |
|--------|--------------------------------------|-----------------|-------------|
| Layout | 4 files (train, pipeline, scheduler, reward) | 1 file dpok_imagereward.py | Same logic, consolidated |
| Sampling | forward_collect_traj_ddim | sample_trajectory() | Same DDIM + η=1 |
| Stored per step | scalar log_prob | noise_pred_old tensor | Recompute ratio at update |
| Log-ratio | Normal.log_prob difference | analytic Gaussian ratio | Mathematically identical |
| KL | ‖ε_θ−ε_ref‖² (no σ² div.) | ‖ε_θ−ε_ref‖²/(2σ²) | Scales β per timestep |
| Loss | −α·Â·clip(ρ)+β·KL | Same + optional VF MSE | Same |
| ε clip | 1e-4 default | 1e-4 → 1e-3 tuned | Tuned for learning |
| Buffer | FIFO 1000, off-policy reuse | Single-round on-policy | Different sample efficiency |
| Grad accum | 12× microbatch 4 | n pairs one backward | Similar effective batch intent |
| VF | Separate 5-iter phase | Joint backward, sep. optimizer | Same role |
| LoRA | LoRACrossAttnProcessor | PEFT on to_q/k/v/out | Modern API |
| Checkpointing | Enabled | Disabled (PEFT grad bug) | Ours trades memory for correctness |
| DDIM steps | 50 | 20 | Faster, fewer transitions |
| CFG | 7.5 train + log-prob | 7.5 train + log-prob | Matched |
| Reward | ImageReward normalized | ImageReward raw score | Scale offset possible |
| Debug artifacts | LoRA + optimizer only | + curves, CSV, snapshot grid | Ours richer |

Defensible deviations are argued in MEETING_WALKTHROUGH.md §4 (analytic ratio,
σ²-normalized KL, CNN value net, per-prompt snapshots).
